# BiliNote 全量 Issue 时间序修复设计

## 背景

本 fork 基于 `JefferyHcool/BiliNote` 的 `master` 提交
`6d67e5a76a2c8da1dd73067943d39021ed137c26`。截至 2026-07-15，上游共有
160 条 open issue，其中 94 条标记为 bug、44 条标记为 enhancement、22 条未分类。
这些 issue 横跨后端、Web 前端、Tauri 桌面端、浏览器扩展、Docker、下载器、
模型供应商和文档，且包含重复、旧版本反馈、缺少复现信息及外部发布条件问题。

本设计用于长期维护个人 fork。它定义全量 issue 的排序、分阶段处理、状态记录、
分支隔离和验证规则，不改变上游 issue 的状态，也不代表个人 fork 可以关闭上游 issue。

## 目标

1. 将全部 160 条 open issue 纳入一个可追踪队列。
2. 严格按 issue 创建时间从新到旧处理。
3. 每 10 条 issue 组成一个阶段，阶段内保持固定顺序。
4. 每条 issue 都形成可审计结论，而不仅限于代码提交。
5. 行为变更采用测试先行，验证通过后再合并到个人 fork。
6. 保持每条 issue 的修改范围独立，便于回滚、同步上游或后续单独提 PR。

## 非目标

- 不批量关闭、编辑或评论上游 issue。
- 不把互不相关的多个问题合并成一个大提交。
- 不为缺少证据的问题进行猜测性修改。
- 不承诺绕过 Firefox、macOS、视频平台或第三方 API 的账号、签名和服务端限制。
- 不修改系统代理或其他系统级网络设置。

## 排序与阶段

队列以 GitHub `createdAt` 降序排序。创建时间相同的条目按 issue 编号降序排序。
阶段大小固定为 10 条。阶段内必须按队列顺序开始分诊；当前条目形成明确处置结论后，
才进入下一条。外部条件阻塞不会让整个队列停滞，但必须先记录阻塞证据和解除条件。

第一阶段顺序如下：

1. `#420` 希望支持读取 B 站评论区内容并参与视频总结
2. `#419` Feature
3. `#417` Docker 部署：DeepSeek 模型列表为空与 whisper 模型下载失败
4. `#416` 增加钉钉直播回放笔记
5. `#415` 桌面端启动时报后端错误
6. `#404` 桌面客户端后端进程退出，退出码为 1
7. `#401` Firefox 无法安装 XPI 扩展
8. `#400` 支持自定义音频识别 AI
9. `#395` requests 2.32.3 泄漏 netrc 凭据风险
10. `#392` macOS 对应 Python 版本问题

第二阶段从下一条创建时间更早的 open issue 继续，后续阶段使用同一规则生成，
直到 160 条全部形成最终处置结论。

## 台账结构

实施阶段创建总台账 `docs/issue-remediation/README.md`，每条 issue 记录以下字段：

- 顺序、阶段、issue 编号、标题和创建时间
- 工作区和类型
- 当前状态
- 最终处置类型
- 复现或核查证据
- 分支和提交
- 验证命令与结果
- 残余风险或解除阻塞所需条件

每个阶段使用 `docs/issue-remediation/stage-NN.md` 保存详细记录，包括正文和评论摘要、
当前版本核查、根因、修改范围、测试证据及阶段回顾。台账只保存必要摘要和链接，
不复制 Cookie、API key、登录凭据或用户日志中的敏感内容。

## 状态与处置

处理状态固定为：

- `queued`：已入队，尚未开始
- `triaging`：正在读取 issue、核查代码和复现
- `implementing`：已有根因和设计，正在实现
- `verifying`：实现完成，正在运行定向及回归验证
- `resolved`：已形成最终处置并记录证据
- `blocked`：依赖当前不可获得的外部条件

最终处置类型固定为：

- `code-fix`：通过代码修复缺陷
- `feature-implemented`：完成功能请求
- `docs-or-config`：通过文档、构建或配置修正解决
- `already-resolved`：当前基线已经解决，补充验证证据
- `duplicate`：与另一条 issue 根因相同，关联到主条目
- `cannot-reproduce`：按 issue 条件和当前基线无法复现，并记录核查范围
- `blocked-external`：需要签名、账号、硬件、第三方服务或发布权限

`duplicate`、`cannot-reproduce` 和 `blocked-external` 同样属于有效处置，但必须有证据，
不能仅凭标题判断。`blocked-external` 在阶段末和全量队列结束时各回查一次。

## 单条 Issue 流程

1. 获取 issue 正文、全部评论、标签、版本、环境和关联 PR。
2. 检查 `master`、必要时检查上游 `develop` 和发布记录，排除已修复或重复问题。
3. 将复现条件缩减为最小案例，避免使用真实凭据和隐私数据。
4. 对行为变更先写会失败的自动化测试，并确认失败原因与 issue 一致。
5. 完成最小实现，不夹带无关重构或其他 issue。
6. 运行定向测试，再运行受影响工作区的回归检查。
7. 更新阶段记录和总台账，附上命令、结果、提交和残余风险。
8. 将独立分支合并到个人 fork 后，才把该条目标记为 `resolved`。

## 分支与提交

每条 issue 使用独立短期分支。即使最终结论是不需要修改代码，也要在独立文档分支中
提交台账和证据更新：

- bug：`fix/issue-<number>-<slug>`
- 功能：`feature/issue-<number>-<slug>`
- 文档、构建或分诊结论：`docs/issue-<number>-<slug>` 或
  `chore/issue-<number>-<slug>`

提交遵循仓库现有 `type(scope): subject` 规范。一条 issue 可以有多个小提交，
但不能包含其他 issue 的修改。存在共同根因时，只在主 issue 分支修改代码；重复 issue
仍使用自己的文档分支更新台账，处置类型记为 `duplicate`，并引用主 issue 的提交。

## 验证矩阵

| 工作区 | 定向验证 | 合并前回归 |
| --- | --- | --- |
| 后端 | 对应 `pytest` 测试或最小复现脚本 | `pytest`，并按需运行 `python -m py_compile` |
| Web 前端 | 相关组件或服务检查 | `pnpm lint` 与 `pnpm build` |
| 浏览器扩展 | 相关 Vitest 或 Playwright 用例 | `pnpm typecheck`、`pnpm test` 与 `pnpm build` |
| Tauri 桌面端 | 后端 sidecar 或 Rust 定向检查 | `cargo check`，发布类问题按目标平台构建 |
| Docker | `docker compose config` 和相关服务构建 | 容器启动、健康检查及受影响接口冒烟测试 |
| 依赖安全 | 无真实凭据的漏洞回归或版本断言 | `pip check`、后端测试及镜像依赖核查 |

若完整验证受本机缺失的硬件、平台、签名或账号限制，必须运行仍可执行的静态、单元和
构建检查，并在台账中明确未验证部分，不能将其描述为已通过。

## 错误与阻塞处理

- 网络请求失败时保留原错误，使用有限重试；不修改系统代理设置。
- issue 信息不足时先检查当前代码、历史提交、发布说明和相似 issue。
- 需要真实 Cookie、API key 或登录态时，使用伪造凭据做安全边界测试；无法模拟的部分
  标记为 `blocked-external`。
- 修改导致无关测试失败时停止合并，先确认是基线问题还是回归。
- 同一 issue 包含多个独立问题时，在阶段记录中拆成子项，但仍保持一个 issue 的
  完成门槛；只有全部子项形成处置结论后才进入下一条。

## 完成标准

单条 issue 只有同时满足以下条件才算完成：

1. 已记录根因或明确的非代码处置理由。
2. 修改范围与 issue 一致，没有夹带无关变更。
3. 已保存定向验证和受影响工作区的回归结果。
4. 已记录分支、提交、残余风险和外部限制。
5. 总台账与阶段记录状态一致。

一个阶段只有在 10 条 issue 全部达到 `resolved` 或有完整证据的 `blocked` 状态后结束。
全量工作只有在 160 条 issue 均有最终处置，且所有 `blocked-external` 已完成最终回查后结束。

## 实施起点

本规格通过后，先生成全量时间序台账，然后为 `#420` 编写独立的功能设计和实施计划。
`#420` 合并并记录验证结果后，按本规格进入 `#419`，依次推进第一阶段。
